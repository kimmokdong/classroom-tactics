# 배틀 엔진 재설계 기획 초안 (롤체 연장전 시스템 포함)

## 0. 시뮬레이터 제작의 진정한 의도와 방향성

> **"시뮬레이터는 억지 수치를 뽑아내는 기계가 아니라, 엔진의 모순을 찾는 거울이다."**

1. **맹목적인 승률 맞추기의 함정 탈피**
   - 기존의 접근 방식은 시뮬레이터 승률(50% 목표)을 억지로 맞추기 위해 시너지 계수를 무한히 펌핑하거나 너프하는 것이었습니다.
   - 그 결과, 피해 감소 99% 나 피해 증폭 900% 같은, 인게임에서는 절대로 허용될 수 없는 "게임 파괴적 수치"가 탄생했습니다.

2. **다양한 덱의 실전 효용성 검증**
   - 우리의 진짜 목표는 **"인게임에서 유저가 어떤 시너지 조합을 선택하든, 상성이나 배치에 따라 충분히 납득할 만한 위력을 발휘하게 만드는 것"**입니다.
   - 롤토체스와 마찬가지로, 각 시너지는 최대 20~40% 내외의 현실적인 수치 안에서 작동해야 하며, 그럼에도 불구하고 유의미한 전투 결과의 차이를 만들어내야 합니다.

3. **시뮬레이터의 올바른 역할: 이상 탐지(Anomaly Detection)**
   - 앞으로 시뮬레이터는 **"현실적인 수치(하드캡 적용)를 넣었을 때 승률이 박살 나는 시너지를 찾아내는 역할"**을 수행합니다.
   - 만약 롤체 수준의 수치(예: 30% 피해 감소)를 넣었는데 특정 시너지의 승률이 20%가 나온다면, 그것은 수치가 부족한 것이 아니라 **배틀 엔진의 메커니즘(예: 타임오버 판정)이 잘못되었음**을 의미합니다.

---

## 1. 현재 엔진의 근본적인 문제점

### 1-1. 타임오버 = 방어 측 패배 (가장 심각한 문제)
보건부(6)가 99% 피해 감소여도 승률이 20%였던 직접적인 원인.
롤토체스는 이 문제를 **"연장전 시스템"**으로 해결합니다.

### 1-2. 피해 계산 순서 불명확
`dmgReduc`(피해 감소)가 방어력 감소 이후에 추가로 곱해지는지, 순서가 정의되지 않은 상태.

### 1-3. 퍼센트 증폭(방송부 등) 발동 여부 불명확
`distAmp: 0.15`였음에도 실질적인 효과가 없어 보이던 현상 → 거리 계산 로직 검증 필요.

## 2. 핵심: 롤체 스타일 연장전 시스템 (Sudden Death)

타임오버로 전투가 끝나지 않을 때 **일방적 패배 판정 대신**, 롤체와 동일한 방식으로 연장전을 발동합니다.

### 연장전 발동 조건
- **기본 전투 시간**: X초 (현재 엔진 기준으로 확인 필요, 대략 30~40초)
- **연장전 발동**: 기본 시간 초과 시 자동 발동

### 연장전 효과
| 효과 | 수치 | 이유 |
|------|------|------|
| 공격 속도 | **+300%** | 전투 템포를 강제로 끌어올림 |
| 주문력 | **+100% 추가 증가** | 스킬 한 방이 치명적이 되도록 |
| 체력 회복 효과 | **-66%** | 힐로 버티는 조합 차단 |
| 보호막 흡수량 | **-66%** | 쉴드 좀비 전략 차단 |
| CC 지속시간 | **-66%** | 기절로 시간 끄는 것 방지 |

### 연장전 종료 조건
- 한쪽 팀이 전멸하면 즉시 종료 → 해당 팀 패배
- 연장전에도 불구하고 전멸하지 않을 경우 (극히 희귀): **무승부** → 연승/연패 수치 0 초기화

### 전략적 함의 (롤체와 동일)
- **연장전 수혜**: 공속이 빨라질수록 유리한 유닛, 단발 고폭 스킬 보유 유닛
- **연장전 피해**: 힐/쉴드 의존 탱커 조합 (회복 66% 감소로 순식간에 녹음)
- **결론**: 음악(힐), 보건부(피해 감소)는 연장전 진입 전에 전투를 끝내야 유리

## 3. 엔진 수정 우선순위

### 🔴 Priority 1: 연장전(Sudden Death) 시스템 추가
**파일**: `js/battleEngine.js`

```javascript
// 연장전 발동 예시 로직
if (elapsedTime >= BATTLE_TIME_LIMIT) {
  // 연장전 플래그 ON
  suddenDeath = true;
  for (const unit of allUnits) {
    unit.attackSpeed *= 4.0;          // +300% = 4배
    unit.ap *= 2.0;                   // +100%
    unit.healEfficiency *= 0.34;      // -66%
    unit.shieldEfficiency *= 0.34;    // -66%
    unit.ccDurationMult = 0.34;       // CC -66%
  }
}
```

### 🔴 Priority 2: 피해 계산 파이프라인 명확화
**파일**: `js/battleEngine.js`

```
원본 피해
  ↓ × dmgAmp         (시너지 피해 증폭)
  ↓ × distAmp        (방송부 거리 보정)
  ↓ × armorPenalty   (방어력으로 감소, 롤체 공식 기준)
  ↓ × (1 - dmgReduc) (보건부 피해 감소, 최대 75% 하드캡 적용)
  ↓ = 최종 피해
```

> [!IMPORTANT]
> `dmgReduc` 하드캡을 **75%** 로 제한합니다.
> 롤체에서도 방어 시너지는 최대 30~40%대입니다.
> 연장전 시스템이 있으면 50%도 충분히 강력합니다.

### 🔴 Priority 3: 공격 속도 하드캡
**파일**: `js/battleEngine.js`
- 평시: 최대 `2.5 attacks/sec`
- 연장전: 제한 없음 (연장전이 빨리 끝나도록)

### 🟡 Priority 4: 거리 계산 검증 (방송부 distAmp)
**파일**: `js/battleEngine.js`
- 방송부 `distAmp`가 실제로 발동하는지 전투 로그 추가
- 포지션 시스템이 없다면 → 랜덤 거리 or 고정 거리 할당 방식 결정

### 🟡 Priority 5: 힐/쉴드 계산 검증
**파일**: `js/battleEngine.js`
- 음악 `tickHeal`이 최대 HP% 기준인지, 현재 HP% 기준인지 확인
- 급식부 `vampBuff` 흡혈이 실제 딜량 기반으로 계산되는지 확인

## 4. 검증 시나리오 (수정 후 테스트)

| 테스트 | 예상 결과 |
|--------|---------|
| 보건부(6) vs 체육(2) | 연장전 진입 → 힐 효율 급감 → 체육팀 역전 or 무승부 |
| 음악(4) 힐 조합 vs 수학(4) 딜 조합 | 평시에 음악 우세, 연장전에 수학 역전 |
| 방송부(7) vs 선도부(2) | distAmp 로그로 거리 보정값 확인 |

## 5. 수정 완료 후 시너지 재밸런싱 원칙

- **수치는 단순하게**: 피해 증폭 최대 45%, 피해 감소 최대 50% 선에서 시작
- **연장전을 염두에 두고 설계**: "연장전에서 이 시너지가 도움이 되는가?" 기준으로 수치 평가
- **시뮬레이터는 방향 확인 도구로만**: 수치 자동 생성 X, 이상 탐지 O
