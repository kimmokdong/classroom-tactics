# 교실택틱스 프롬프트 히스토리 (Prompt History)

이 문서는 '교실택틱스' 프로젝트 개발 중 발생한 핵심 노하우, 시행착오 및 해결책을 기록하여 향후 유사 작업 시 프롬프트 지식 자산으로 활용하기 위해 작성되었습니다.

## [2026-05-23] 스킬 툴팁 동적 렌더링 및 텍스트/수치 분리 로직 구현
**문제 (Challenge):** 
유닛의 스킬 설명에 `주문력(150% / 250% / 400%) 비례 보호막`과 같이 성급별 수치가 하드코딩되어 있어 가독성이 떨어졌으며, 게임 중 아이템 착용 등으로 변동되는 실시간 스탯(AP, AD)이 텍스트에 반영되지 않아 불편함이 존재했습니다.

**해결 (Action):**
1. **데이터 계층 분리:** `data.js`의 `skill.desc`에는 순수 텍스트(예: `가까운 아군 2명에게 보호막 부여`)만 남기고 수치는 완전히 제거했습니다.
2. **UI 렌더링 시 정규식 동적 주입:** `main.js`의 `formatSkillDesc()`에서 `html.replace()` 정규식을 활용해 텍스트를 파싱하고, 현재 유닛의 스탯(currAp, currAd)이 반영된 동적 수치와 스탯 아이콘(🔮, ⚔️ 등)을 주입하도록 구현했습니다.

**교훈 (Pattern):**
- **"데이터는 심플하게, UI 렌더링은 똑똑하게"** 
- RPG/Auto-battler 게임에서 툴팁을 설계할 때는 원본 데이터(JSON)에 텍스트와 변동 수치를 하드코딩으로 섞지 마십시오. 문자열을 깔끔하게 유지하고 클라이언트(UI 렌더링 함수) 측에서 정규식을 통해 변수를 주입하는 구조가 확장성 측면에서 가장 유리합니다.

## [2026-05-23] UI 스탯 툴팁 색상 통일 및 아이콘 표기 철학 정립
**문제 (Challenge):** 
유닛 상세 정보 패널, 상점 툴팁, 유닛 도감 등 띄워지는 UI마다 체력/마나/방어력 스탯의 색상이 일치하지 않아 혼동을 주었습니다. 또한, AP/AD 스케일링을 받지 않는 '고정 수치(Flat Value)'임에도 불구하고 색상과 아이콘이 붙어 있어 유저가 "스케일링이 되는 수치"로 오해하는 문제가 발생했습니다.

**해결 (Action):**
1. **단일 색상 객체(Single Source of Truth):** `COLORS` 객체(`hp: '#4caf50'`, `ad: '#e65100'`, 등)를 정의하고 모든 컴포넌트(도감, 상점, 정보창)에 동일한 헥스 코드를 적용했습니다.
2. **시각적 규칙 확립:** AP/AD 계수의 스케일링을 받는 수치에만 해당 스탯 아이콘(🔮, ⚔️)과 색상을 표기하고, 마나 소멸, 방/마저 퍼센트 감소 등 스케일링과 무관한 **고정 수치/비율**에는 기본 텍스트 색상(검은색, `#333333`)과 숫자만 표기하여 오해를 원천 차단했습니다.

**교훈 (Pattern):**
- **"색상과 아이콘은 곧 시스템 메커니즘을 의미한다."** 
- 툴팁에서 텍스트 색상이나 스탯 아이콘을 남발하지 마십시오. 아이콘과 색상은 반드시 유저가 "이 스탯을 올리면 이 효과가 강해진다"는 것을 인지할 때만 사용해야 합니다. 고정 수치는 기본 색상으로 무미건조하게 표기하는 것이 UX/UI의 정석입니다.

## [2026-05-23] addBuff 버그: 스킬 버프가 실제 스탯에 반영되지 않던 치명적 설계 결함
**문제 (Challenge):**
`addBuff(target, type, stat, val, duration)` 함수가 `buffs` 배열에 버프 객체를 push할 뿐, `target.stats[stat] += val`을 실행하지 않았습니다. 그 결과, 전투 중 발동된 모든 스킬 버프(아군 AP 증가, 공속 증가, 방어력 증가 등)가 실제 전투 계산에 전혀 반영되지 않는 치명적 버그가 있었습니다. 또한 버프 만료 시점에도 스탯 복구 로직이 없었습니다.

**해결 (Action):**
1. **즉시 스탯 적용:** `addBuff` 함수에 `else if (stat && target.stats[stat] !== undefined) { target.stats[stat] += val; }` 조건을 추가하여 `ad`, `ap`, `as`, `armor`, `mr`, `range` 등 스탯이 즉시 반영되도록 수정.
2. **만료 시 원상복구:** 버프 tick 처리 루프(`buffs.splice`)에서 만료된 버프의 `expired.stat`이 스탯 키에 해당하면 `u.stats[expired.stat] -= expired.val`로 차감하여 정확한 상태 복구 구현.
3. **vampBuff 별도 처리:** `vamp`는 `stats` 객체에 없고 `combat.vamp`로 관리되므로, `addBuff` 호출 대신 `unit.combat.vamp += val`로 직접 적용하고 `vampBuffTimer`로 별도 지속시간 관리.

**교훈 (Pattern):**
- **"push만 하고 쓰지 않는 배열은 버그다."**
- `buffs` 배열에 값을 push해도 그 값을 전투 계산 시점에 읽는 코드가 없으면, 버프는 실질적으로 무효입니다. 버프 시스템 설계 시에는 반드시 ① 적용 시점(즉시 stats 반영), ② 만료 시점(stats 복구)을 쌍으로 구현해야 합니다.
- `vamp`처럼 `stats` 객체가 아닌 `combat` 객체에서 관리되는 속성은 일반화된 `addBuff`로 처리할 수 없으므로 별도 로직으로 분리해야 합니다.

## [2026-05-23] 사회 시너지 활성화 로직 리디자인 (정확한 인원 매칭 기믹)
**문제 (Challenge):**
기존 사회 시너지는 `dmgAmp(1) / shield(2) / allStats(4)` 순서로 설계되어 고립(독불장군)의 컨셉과 어울리지 않았고, 2나 3인 상황에서도 1시너지 버프가 발동하는 구조적 문제가 있었습니다. "오직 1명이거나, 4명 완전체일 때만" 발동하는 특수 조건(TFT의 추방자나 닌자 기믹과 유사)이 필요했습니다.

**해결 (Action):**
1. **데이터 스키마 추가:** `data.js`의 사회 시너지 속성에 `exactMatch: true` 플래그를 추가하고, `levels: { 1: { shield: 300 }, 4: { allStats: 0.3 } }`로 정리했습니다.
2. **시너지 레벨 산출 로직 개편:** `main.js`의 `getActiveSynergyLevel` 및 `getLevelData` 함수에서 `exactMatch` 플래그가 주어질 경우, `count >= lvl` 검사가 아닌 `levelsArr.includes(String(count))`를 통해 개수가 **정확히 일치할 때만** 활성 레벨로 반환하도록 수정했습니다.
3. **UI/상태 반영 연동:** 적군 UI 산출 로직과 아군 툴팁/시너지 렌더링 로직 모두에서 해당 플래그를 넘겨주도록 일괄 업데이트하여 2명이나 3명일 때는 UI에서도 비활성화 상태(회색 텍스트/보더)가 되도록 처리했습니다.

**교훈 (Pattern):**
- 시너지 발동 조건은 단순한 "이상(>=)" 조건뿐만 아니라, 컨셉에 따라 "정확히 일치(==)"하는 경우도 대비해야 합니다. 이를 위해 공통 함수(`getActiveSynergyLevel`)에 선택적 매개변수(`exactMatch = false`)를 도입하여 유연한 기믹 적용이 가능하도록 설계하는 것이 바람직합니다.

## [2026-05-24] 전투 데미지 텍스트 겹침 방지(Scatter) 및 속성별 렌더링
**문제 (Challenge):**
다수의 유닛이 한 대상을 점사할 때 데미지 텍스트가 똑같은 좌표(0,0)에서 렌더링되어 한 개의 숫자처럼 보이는 가시성 문제가 있었습니다. 물리/마법/고정/회복/치명타 등 다양한 전투 정보도 구분이 안 되었습니다.

**해결 (Action):**
1. **로그 타입 세분화:** `battleEngine.js`에서 데미지 계산 시 `this.logs.push`에 `dmgType`(physical, magic, true) 및 `isCrit`, 회복(`type: 'heal'`)을 함께 명시하도록 개선했습니다.
2. **Scatter 오프셋 적용:** `battleRenderer.js`에서 텍스트 엘리먼트를 생성할 때 `Math.random()`을 사용해 `marginLeft`와 `marginTop`에 -25px ~ +25px 랜덤 오프셋을 부여해 사방으로 흩뿌려지게 만들었습니다. CSS `transform` 애니메이션과 충돌하지 않기 위해 `margin` 속성을 활용했습니다.

**교훈 (Pattern):**
- 플로팅 데미지 텍스트가 여러 개 동시 발생할 수 있는 게임에서는 반드시 Scatter(흩뿌리기) 로직이 필수입니다. 이때 CSS Keyframe이 `transform`을 쓰고 있다면, 충돌을 피하기 위해 `margin`이나 `left/top` 속성으로 오프셋을 주어야 합니다.

## [2026-05-24] 대기석 유닛 자동 배치 시 UI 동기화 누락 수정
**문제 (Challenge):**
전투 시작(`btn-start-battle`) 시 보드에 남는 자리가 있으면 대기석 유닛이 자동으로 올라가지만, 좌측 시너지 UI가 갱신되지 않아 전투 중 실제 발동 중인 시너지와 표기되는 시너지가 불일치했습니다.

**해결 (Action):**
자동 배치 로직이 끝나는 블록 직후에 `app.calculateSynergy()` 호출 한 줄을 추가하여 state 변경과 화면 렌더링을 완전히 동기화했습니다.

**교훈 (Pattern):**
- **상태(State)를 강제로 변경했다면 연계된 모든 계산 함수를 명시적으로 호출해야 합니다.** 화면 렌더링(`renderUnits()`)만 호출했다고 해서 파생 데이터(Synergy)가 자동으로 업데이트되는 것은 아닙니다.
