## [2026-05-31] 캔버스 파티클 효과와 DOM UI 동기화 및 타겟팅 좌표 역산출
**문제 (Challenge):**
기부천사 패시브 발동 시 아군에게 상자가 떨어지는 시각 효과를 적용하는 과정에서 여러 연쇄적인 오류가 발생했습니다.
1. **DOM 팝업 증발:** 처음에 `div`를 생성해 CSS 팝업 애니메이션을 시도했으나, 렌더링 엔진(`renderBoard`)이 틱마다 화면을 다시 그리면서 DOM 요소가 즉시 증발했습니다.
2. **이펙트 엔진 명령 무시:** 캔버스 파티클 시스템(`FxSystem.js`)으로 전환했으나, `spawnFx`에 해당 이펙트 식별자 분기가 누락되어 화면에 그려지지 않았습니다.
3. **타겟 좌표 및 렌더링 타이밍 오차:** 대상의 정확한 위치(발 밑)를 찾기 위해 `getBoundingClientRect`를 썼지만, 화면 렌더링 직후인 0ms 시점이라 레이아웃이 미완성되어 좌표가 중앙(0,0)으로 튀는 문제가 있었습니다.
4. **인덱스(Index) 기반 좌표 추출 버그:** 보드 배열(Array) 인덱스와 실제 DOM 화면(`querySelectorAll`)의 배치 순서가 달랐습니다. 샌드박스의 경우 0~23 타일이 적군, 24~47이 아군인데 아군 배열 인덱스 `[i]`를 그대로 DOM에 매핑하여 이펙트가 적군 진영에 떨어지는 치명적 버그가 발생했습니다.
5. **UI 동기화 및 텍스트 렌더링 하드코딩:** 캔버스 텍스트 이펙트(`low_passive`)가 하드코딩된 노란색상과 빠른 페이드아웃 때문에 회색으로 묻혀버렸고, 샌드박스와 본게임 간 아이템 장착 UI 렌더링 클래스(`.unit-item-overlay`) 및 로직이 파편화되어 이펙트는 터지는데 아이템 장착 UI가 보이지 않는 문제가 있었습니다.

**해결 (Action):**
1. **렌더링 위임:** 일회성 CSS DOM 이펙트 대신, 모든 파티클을 캔버스 렌더러(`FxSystem.js`) 파이프라인에 등록하여 엔진 생명주기와 동기화했습니다.
2. **타이밍 지연 및 인덱스 오프셋(Offset) 적용:** 보드가 완전히 그려지고 DOM 배치가 끝난 시점을 보장하기 위해 `setTimeout(..., 150)`을 적용했습니다. 또한 샌드박스와 본게임에서 아군 필드를 찾기 위해 DOM 탐색 시 `[i + 24]` 오프셋을 명시하여 정확한 아군 DOM 엘리먼트의 절대 좌표를 추출해냈습니다.
3. **텍스트 이펙트 고도화:** `FxRenderer.js`의 `low_passive` 텍스트 로직에서 하드코딩을 걷어내고, 색상을 검정(`#000000`), 굵기 `900`, 흰색 스트로크 `lineWidth: 3`으로 변경하여 가독성을 극대화했습니다. 
4. **UI 클래스 단일화:** 아이템 렌더링 로직을 본게임의 아이템 슬롯 규격(`.unit-item-overlay`, 우측 상단 절대좌표)으로 통일하고, 샌드박스의 `setTimeout` 내부에도 동일한 UI 갱신 코드를 강제로 심어 효과와 정보창(UI)을 동기화시켰습니다.

**교훈 (Pattern):**
- **DOM 레이아웃이 확정되기 전에 좌표계를 요구하면 무조건 0이나 엉뚱한 위치가 반환된다.** 좌표 기반 파티클을 그릴 때는 렌더링 스레드가 화면을 갱신할 미세한 유예 시간(`setTimeout`)이 반드시 필요합니다.
- **논리적 배열(데이터)과 물리적 DOM(화면 요소)의 인덱스는 일치하지 않을 수 있다.** 아군/적군/대기석이 혼합된 렌더링 화면에서 `querySelectorAll`로 타겟을 역추적할 때는 오프셋(Offset)이 정확한지 반드시 검증해야 합니다.
- **시각 엔진(Canvas) 내의 하드코딩은 외부의 파라미터 주입을 씹어버린다.** 파티클 색상이나 투명도 로직은 범용적으로 활용할 수 있게 외부 파라미터(`p.color`) 우선 정책과 세밀한 조건 제어를 적용해야 합니다.
- **시각효과의 완전한 파이프라인 관통 구조 이해:** 오토배틀러나 RPG 등에서 시각효과 하나를 화면에 완벽하게 띄우기 위해서는 **(1) `StageManager` / `sandbox` (발동 트리거 및 좌표 계산) -> (2) `FxSystem` (이펙트 식별 및 캔버스 파티클 정의) -> (3) `FxRenderer` (정의된 파티클을 캔버스에 실제로 그리는 로직)** 이라는 3단계 파이프라인 모두에 정확하게 코드를 심고 연동시켜야만 누락 없이 완전하게 동작함을 명심해야 합니다.

## [2026-05-30] HTML5 드래그 앤 드롭 자식 요소 dragleave 버그 해결 및 미리보기 툴팁 구현
**문제 (Challenge):**
인벤토리에서 아이템을 드래그하여 다른 조합 템 위에 올렸을 때(dragover), 어떤 상위 아이템으로 조합되는지 미리 보여주는 툴팁 기능을 추가했습니다. 하지만 슬롯 내부에 있는 아이콘 이미지(자식 요소) 위로 마우스가 지나갈 때마다 슬롯의 `dragleave` 이벤트가 발생하여 툴팁이 무한히 깜빡이거나 즉시 사라지는 고질적인 HTML5 버그가 발생했습니다.

**해결 (Action):**
단순히 `dragleave`에서 툴팁을 끄는 대신, **`dragCounter` 패턴**을 도입했습니다.
슬롯에 `dragenter` 할 때마다 카운터를 +1 하고, `dragleave` 할 때마다 -1 하도록 설정한 뒤, **`dragCounter === 0`이 될 때만 툴팁을 숨기도록(hideCustomTooltip) 로직을 수정**했습니다. 이를 통해 자식 요소 진입/이탈 시 발생하는 불필요한 이벤트 버블링을 완벽하게 무시했습니다.

**교훈 (Pattern):**
- **"HTML5 Drag and Drop에서 부모의 dragleave는 자식 진입 시에도 발생한다."**
- 드롭 영역(Dropzone) 내부에 자식 요소가 존재할 경우, `pointer-events: none`으로 자식을 막거나 **`dragCounter`를 활용해 진입/이탈 횟수의 밸런스가 0이 될 때만 완전한 이탈로 간주하는 패턴**을 적용해야 툴팁 깜빡임이나 드롭 판정 오류를 원천 차단할 수 있습니다.

**교훈 (Pattern):**
- 캔버스 기반 루프가 하나라도 크래시나면 `requestAnimationFrame` 스케줄링이 중단되면서 이전 화면(특히 플래시나 오버레이)이 굳어버리므로 **렌더링 루프 전체에 try-catch 에러 방어막을 씌우고 모달 종료/전환 시 리셋(클리어) 처리를 반드시 보강**해야 합니다.

- 지연 전파되는 장판(독액 등)은 개별 파티클에 **시작 대기 시간(maxT) 오프셋을 부여하고, maxT를 넘은 순간부터 크기를 확장(scale)하는 방식**으로 코딩하면 복잡한 연쇄 트리거 없이도 환상적인 파동 전파 시각 효과를 낼 수 있습니다.

## [2026-05-23] addBuff 버그: 스킬 버프가 실제 스탯에 반영되지 않던 치명적 설계 결함
**문제 (Challenge):**
`addBuff(target, type, stat, val, duration)` 함수가 `buffs` 배열에 버프 객체를 push할 뿐, `target.stats[stat] += val`을 실행하지 않았습니다. 그 결과, 전투 중 발동된 모든 스킬 버프(아군 AP 증가, 공속 증가, 방어력 증가 등)가 실제 전투 계산에 전혀 반영되지 않는 치명적 버그가 있었습니다. 또한 버프 만료 시점에도 스탯 복구 로직이 없었습니다.

**해결 (Action):**
1. **즉시 스탯 적용:** `addBuff` 함수에 `else if (stat && target.stats[stat] !== undefined) { target.stats[stat] += val; }` 조건을 추가하여 `ad`, `ap`, `as`, `armor`, `mr`, `range` 등 스탯이 즉시 반영되도록 수정.
2. **만료 시 원상복구:** 버프 tick 처리 루프(`buffs.splice`)에서 만료된 버프의 `expired.stat`이 스탯 키에 해당하면 `u.stats[expired.stat] -= expired.val`로 차감하여 정확한 상태 복구 구현.
3. **vampBuff 별도 처리:** `vamp`는 `stats` 객체에 없고 `combat.vamp`로 관리되므로, `addBuff` 호출 대신 `unit.combat.vamp += val`로 직접 적용하고 `vampBuffTimer`로 별도 지속시간 관리.

**교훈 (Pattern):**
- **"push만 하고 쓰지 않는 배열은 버그다."**
- `buffs` 배열에 값을 push해도 그 값을 전투 계산 시점에 읽는 코드가 없으면, 버프는 실질적으로 무효입니다. 버프 시스템 설계 시에는 반드시 ① 적용 시점(즉시 stats 반영), ② 만료 시점(stats 복구)을 쌍으로 구현해야 합니다.
- `vamp`처럼 `stats` 객체가 아닌 `combat` 객체에서 관리되는 속성은 일반화된 `addBuff`로 처리할 수 없으므로 별도 로직으로 분리해야 합니다.
